# Mosh JD — Sitemap & Indexability Review

**Prepared for:** Wellington Coelho / Mosh JD
**Date:** July 23, 2026
**Scope:** Audit the sitemap, resolve the Semrush "2,490 pages vs ~100 real pages" discrepancy, optimize, and resubmit to Google Search Console.

---

## Bottom line

**There is no indexability problem and no backend bug generating "ghost pages."** The website's sitemap is clean and correct, and Google is indexing the real pages exactly as it should. The 2,490 figure in Semrush was a **crawler-scope artifact**: Semrush's Site Audit was set to crawl the entire domain *including subdomains*, so ~90% of what it counted was the **newsletter (`newsletter.moshjd.com`, a beehiiv platform)** — not the website. We corrected the audit configuration; the report now reflects the real site.

---

## What we found

**1. The sitemap is healthy.**
- Located at `https://www.moshjd.com/sitemap.xml` — exists, valid XML, generated by All in One SEO Pro (WordPress).
- Contains **193 real, unique URLs**: 131 blog posts, 59 pages, 3 testimonials.
- No duplicates, no broken links, no parameter/ghost URLs, consistent `https://www.` host.
- **`app.moshjd.com` (login/app) is NOT in the sitemap** — correct.

**2. The "2,490 pages" was the newsletter subdomain, not the site.**
Using Semrush's own crawled-pages export (2,671 URLs), the breakdown by hostname:

| Source | URLs | Share |
|---|---|---|
| `newsletter.moshjd.com` (beehiiv newsletter) | 2,402 | **90%** |
| `www.moshjd.com` (the actual website) | 252 | 9% |
| `moshjd.com` (non-www redirects) | 14 | <1% |
| `app.moshjd.com` (product app) | 3 | <1% |

Semrush's own data confirms Google won't index the excess: only **204 URLs are unique real pages** ("self-canonical"); **2,412 are duplicates** that point their canonical tag to the real page (Google consolidates these). The alarming "7,730 warnings" spike was almost entirely the beehiiv pages (missing meta descriptions / H1s that the newsletter platform doesn't set).

**3. Google is fine.** Because the sitemap is clean and the site returns correct signals (proper 404s, `noindex` on search/feed/utility pages, archives disabled), Google indexes only the ~100–200 real pages. The gap between Semrush (2,490) and Google (~100) is **scope, not a defect.**

---

## What we changed

**Reconfigured the Semrush Site Audit** so the report measures the website, not the newsletter:
- Scope set to `www.moshjd.com` with **"Crawl all subdomains" turned OFF** (removes the beehiiv subdomain).
- Crawl source changed to **the sitemap** (crawls the real ~193 URLs instead of chasing every internal link).
- Re-ran the audit.

**Result (re-run completed July 23, 2026):**

| Metric | Before | After |
|---|---|---|
| Pages crawled | 2,670 | **201** |
| — of which the newsletter subdomain | 2,402 | **0** |
| — parameterized/tracking URLs | 2,400 | **0** |
| Warnings | 7,730 | **330** (−7,400) |
| Errors | 115 | **22** (−93) |
| Site Health | 76% | **84%** (+8%) |

The re-crawled set is now **200 `www.moshjd.com` pages + 1 non-www redirect — zero newsletter, zero app, zero parameter URLs**, exactly matching the real site.

**The sitemap itself required no changes** — it was already well-optimized. (Note: All in One SEO deliberately omits `<priority>`/`<changefreq>` tags because Google ignores them, so those are not applicable to this setup.)

---

## Flags that need attention

1. **`app.moshjd.com` should be set to `noindex`** — the app subdomain returns HTTP 200 for any URL and is fully open to crawlers. It's not causing the Semrush number, but it's poor SEO hygiene. This is a **development-agency change** (separate app, not WordPress).
2. **Genuine site-level cleanup (WordPress / AIOSEO / Elementor):** now that the noise is gone, the real audit shows a small, manageable issue list worth a cleanup pass:
   - **20 invalid structured-data items** (Error) — likely theme/Elementor schema; validate and fix.
   - **62 pages where the H1 duplicates the title tag** — differentiate H1 vs SEO title in AIOSEO.
   - **23 page titles too long** — trim to ≤60 chars.
   - **21 pages missing an H1** — add a proper H1 (often Elementor pages using styled headings that aren't H1).
   - **191 pages with low text-HTML ratio** — common on design-heavy Elementor pages; usually low priority.
3. **Newsletter indexing** — the beehiiv pages are real and indexable (usually desired). No action unless the client wants the newsletter excluded from search, which is a beehiiv-side setting.

## Google Search Console — confirmed

The Coverage/Pages export (filtered to the submitted `sitemap.xml`, data through Jul 9) settles the client's question with Google's own numbers:

| Metric | Value |
|---|---|
| Indexed | **126** |
| Not indexed | **73** |
| **Total** | **199** — matches the real site (Semrush re-crawl: 201; sitemap: ~193) |

- **No indexability issue.** Google sees ~199 pages, nowhere near 2,490.
- The 73 not-indexed are benign: **48 "Discovered – currently not indexed"** (queued / crawl priority — internal linking helps) and **25 "Crawled – currently not indexed"** (thin/low-value — the noindex + content-optimization candidates).
- The sitemap is **registered and actively used** by Google (it's the source of the 126 indexed pages). A fresh "Success" screenshot on resubmission is optional confirmation; not required to answer the question.

Yes. The discrepancy was diagnosed to its true root cause (crawler scope + a newsletter subdomain, not a site bug), verified against Semrush's own raw data, corrected at the source, and the sitemap confirmed healthy — without any unnecessary changes to a site that didn't need them.
